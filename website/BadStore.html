General Info

Unlike DVWA, Bad Store is a virtual machine hosting a extremely vulnerable web site.
Just like DVWA, it is also publicly available and designed to be vulnerable.

I have chosen not to attach Bad Store onto the same virtual machine to help clean up the Nmap response (and to try and simulate penetration testing an external target).

Bad store is a simple starting point for learning how to attack websites, some of the vulnerability’s Bad Store has to offer are more non-technical, with some vulnerabilities being caused due to bad practices.

Whenever you are using BadStore, you MUST run the BadStore VM at the same time. Otherwise you cannot find it. The BadStore folder can be found in the same location as the Penting folder.

Setting up Bad Store

In case this is your first time setting up Bad Store and you cannot connect to it, click here to see the instructions in case Bad Store is not working

If you are using Vbox, you will need to remake the BadStore VM. I have placed the BadStore ISO in the BadStore folder.
Warning: reading these instructions may spoil the answers for some of the tasks:

  1. First run both BadStore VM and Penting VM
  2. Once on, check the IP address of Bad Store. This can be done using ifconfig. 
     ifconfig
  3. Then in Penting, go to the terminal try to ping the IP address of BadStore
     ping IP

  4. If you are getting responses from the IP then it means you are connected. You can also double check it works by then trying the IP out on the website, though I wouldn’t because it may spoil the tasks.

1. After turning on the Bad Store virtual machine, find out what its IP address is

First I got the IP address of the Penting VM using ifconfig

Then with the IP address of Penting and nmap command I did two nmap host discovery scans – one in which BadStore was turned off, another when it was turned on. This made it so the Bad Store IP would show up and be identified in the second scan. 

 nmap -sn IP.0/24

2. Using Nmap, scan the IP address revealed in task 1 and identify what port the web service is running on and its version.
Then proceed to access the website through the browser

First I did a nmap -p- IP -A to scan all the ports and get full information.

 nmap -p- -A IP


As you can tell the service version is very out of date.

3. Using nikto, scan the IP addresses web service found. Name the ‘interesting’ directories found

For this task, just doing nikto -host IP is enough

 nikto -host IP


The interesting directories found are the backup and supplier directories. Right now I will not do anything with them, but it’s best to be aware of them for later.

4. What page revealed the location of these directories, and how do you access that file? Also highlight where the issue is.

The page that mostly revealed the directories is robots.txt (files, like backup, would have been picked up by nikto anyway but because it was stored in here it was found much easily). If the supplier directory however was not in robots.txt then it may not have been detected by nikto.

The page can be accessed through the browser by adding ‘robots.txt’ to the IP address. Many websites have a robots.txt file, it’s used to help the site get scanned by Google.

The issue is the disallow section. The directories are listed here to stop scan bots from looking at these directories, however that doesn’t stop us from looking at them.

5. Take a look at each of the interesting directories, one of them should have an even more interesting file. What is its name?

In the supplier directory a file called ‘accounts’ can be found

Within the accounts file hash values can be found

Can you tell what file it resembles?

The hash is base64. The site I used to identify the has ‘hashes.com’.

The decryption of the hashes can be found below. These look like credentials from a Linux passwd file. The site used for decryption was ‘CyberChef’.

7. Now to look at the website. Enter the Bad Store IP address into the URL.
Once in, navigate to the 'my account' page and read the information regarding the password reset system. What do you notice?

In order to change a users password, you only need the email address and a password hint. Not too bad so far but…

The password hint only provides a selection of 6 choices, as opposed to using a user generated answer. This makes it susceptible to brute force attacks (an attacker could use this list to reset a list of account passwords).

8. After looking at the page, try to reset a random user’s password.
What do you notice about the result of the password reset?

Regardless of what the user’s password hint and username was (look at the username I entered, it most definetly does not exist), the password will always be reset to ‘Welcome’. This is a MAJOR flaw in the system, this meant by just knowing the username (and guessing the hint) a random person could reset another users account.

18.1 Depending on the username you entered, you will receive responses of different length. Why are these length different?

The reason for the different length is solely because of the size of the email address/username added (see above). For example, look at the difference of the length between when payload 1 used root and when it used admin. Root is one letter shorter than admin, and as such it always has a length shorter than admin by one.
