General Info

DVWA (Damn Vulnerable Web Application) is a publicly available web application designed to be vulnerable. It is a good starting point when it comes to learning how to use tools against an intentionally vulnerable service.
This page will provide activities to undergo against the DVWA website, mostly looking at Nmap, Burp Suite and Nikto.

Usually web services are hosted on port 80 or port 443. On this virtual machine I have changed the port location for DVWA. Your first task is to find the port and access the DVWA site.
As a freebie hint, when you find what port DVWA is running on add a colon after the IP address with the port number like so

In this case, finding the service version is of little relevance, however if you are dealing with an out of date service it will be important. This is because if it is of a certain version it may have certain vulnerabilities. Famous examples include the Log4j vulnerability, Heartbleed and shellshock.

1. Find the DVWA port, the site and state the server version

The port can be found by adding the –p- argument to the nmap scan.

 nmap -p- localhost


Doing this however does not reveal the service, nor specifically which one is DVWA

Adding –A to the command provides the additional information needed. It does however also produce a wall of text, but near the bottom of the information, info regarding DVWA can be found, including the service version.

The Apache server version was 2.4.41.

1.1. (optional) Find the Operating Systems versions EternalBlue worked against.

The main target of the EternalBlue are a range of Windows Machines, 2016 all the way to 2000. HTB has a machine called ’Blue’ that is based off the EternalBlue exploit, it is a fairly simple box to exploit. TryHackMe also has a free alternative, also just called ‘Blue’.
Here is the list to the Exploitdb site of an exploit found from just looking off Google: https://www.exploit-db.com/exploits/42315

2. What is the DVWA login credentials?

The username is admin, the password is password. It was fine to use google to get the credentials, especially since I used the default credentials for this purpose. In many companies default credetnials for services and admin accounts are often still used.

3. When logged in, change the security level in DVWA to low.

Navigate to the DVWA security tab, select the drop down and select 'low'.

Besides the need to adjust the DVWA settings every time it resets, some systems actually work like this when it comes to system security. This is why its important to ensure the admin account remains secure and not have it compromised in the first place.

4. On the DVWA site a interesting directory has been hidden. Using Nikto, can you identify the folder name?

Nikto with the host and port specified is enough to get the job done

 nikto -host 127.0.0.1 -port 45080

From the results, a folder called ‘secret’ can be found. This is a folder I have made. If you find one out in the wild, it will most likely contain information you’re not supposed to see!

The other folders, tests and database are also very valuable. Tests may contain sensitive documents or forgotten credentials, such as a long lost test user with admin credentials. As it is an ‘experimental’ user it may have more permissions then the normal user. In this case it contains a file used to trim down on waste. Database files also have the same value as well.

5. Access the interesting folder (found in task 4) and download the wordlists present

First I went into the area of the interesting directory and found two wordlist files.

I then proceeded to copy the URL to this page and added the file name to the end of it, so it looked like this.

I then copy and pasted the URLs into a curl command and added the –o operator to save the file on the VM.

curl localhost:45080/secret/passwords.txt -o passwords.txt  
curl localhost:45080/secret/usernames.txt -o usernames.txt

With that I then got the files on my VM to be used as a wordlist.

11.1. In the HTTP request, an interesting parameter, a cookie, can be found. What is the name of the parameter, and how would we add it as a payload position?

Under ‘attack type’, read the type of attacks available and select the attack suitable for brute forcing the login page.

The attack being done is called a ‘Cluster Bomb’.

This attack type is being used because two separate payloads (wordlists) are being used for brute forcing – the username and password list. Sniper attack could be used however it would take too long – it would go through all the payload combinations (and so try the password wordlist in the username field, which would take up time).

21. Return to the DVWA page, enter a SQLi query into the ID field to present all the user IDs on the page (make sure intercept is turned off)

Enter 1' or 1=1# into the ID area like so, and the following IDs will be seen

Broken down, the apostrophe (') closes out the SQL query early. What follows next is a True statement (or 1=1), which will always be true.
This results in the database then reporting back all user IDs. The hash (#) is used to escape the statement early, cutting the rest of the query off.
